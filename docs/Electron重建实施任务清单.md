# YiboBattleSwitch Electron 重建实施任务清单

## 1. 文档目的

本清单用于把 [Electron重建方案.md](E:/Program/YiboBattleSwitch/docs/Electron重建方案.md) 转化为可执行的实施路线。

目标是让后续重建工作具备以下特征：

- 能按阶段推进
- 每阶段都有明确产出
- 每阶段都有验收标准
- 能持续校验“是否仍然在彻底摆脱 NewBeeBox 运行时包袱”

本清单只针对正式 Electron 重建主线，不针对历史原型继续扩展。

## 2. 总体实施原则

### 2.1 先搭新壳，再迁能力

禁止一边沿用旧大包，一边零散迁移功能。  
必须先建立新的 Electron 工程骨架，再逐步接入能力。

### 2.2 先保可运行，再保可切换

第一优先级是新应用可稳定启动、显示、通信、落日志。  
第二优先级才是账号切换、备份恢复、诊断。

### 2.3 先普通权限闭环，再考虑提权扩展

重建首期必须先把普通权限下可行的 Battle.net 进程治理做完整。  
不把高权限 helper 当成首期前提。

### 2.4 先剥离 NewBeeBox 运行时依赖，再保留导入能力

任何功能接入前都要先问：

- 这是本程序自有能力吗
- 还是在继续借用 NewBeeBox 运行时

只允许保留“导入历史数据”这一个产品级关系。

## 3. 阶段总览

建议按 9 个阶段推进：

1. 新工程落地
2. 前端 UI 迁入与本地化
3. Electron 基础能力接入
4. 自有存储体系建立
5. Battle.net 基础能力接入
6. 普通权限进程治理落地
7. 正式切换事务落地
8. 导入与诊断能力迁移
9. 安装包与发布链落地

## 4. 阶段 1：新工程落地

### 4.1 目标

建立一个全新的、干净的 Electron 应用骨架，完全不依赖历史 `electron-main.js`。

### 4.2 任务

- 新建 Electron 主工程目录
- 建立 `main / renderer / domain / infra / shared / assets` 基础目录
- 接入 TypeScript
- 建立基础构建脚本
- 建立开发启动脚本
- 建立 lint / format / typecheck 基础命令
- 建立 `.env` 或配置分层约定
- 建立基础日志目录与日志初始化
- 明确不引用根目录现有 `[electron-main.js](E:/Program/YiboBattleSwitch/electron-main.js)`

### 4.3 产出

- 新工程可启动
- 空白主窗口可显示
- 主进程和渲染进程日志可落盘
- 单实例限制可生效

### 4.4 验收标准

- 启动时不读取 `NewBeeBox` 任何目录
- 启动时不读取旧 `electron-main.js`
- 主窗口能稳定显示，不白屏
- 控制台和文件日志都能看到应用启动记录

### 4.5 风险与注意事项

- 不要在这个阶段提前接 Battle.net 业务
- 不要从旧 Electron 包里复制大段主进程逻辑

## 5. 阶段 2：前端 UI 迁入与本地化

### 5.1 目标

在新 Electron 工程中显示当前产品 UI，并去掉所有外部资源依赖。

### 5.2 任务

- 迁入当前前端页面结构
- 迁入必要样式、图标、字体资源
- 将 CDN Tailwind 改成本地构建产物
- 将 Google Fonts 改为本地字体资源
- 建立 `preload` 与前端桥接结构
- 清理原型中的本地 HTTP 调用方式
- 建立统一前端 API 封装层

### 5.3 产出

- 现有 UI 在 Electron 中可显示
- UI 首屏只依赖本地资源
- 前端已具备调用 `window.api.*` 的基础结构

### 5.4 验收标准

- 断网情况下 UI 可完整显示
- 启动时不访问外网字体、脚本、样式
- 页面可以渲染主布局、按钮、账号列表占位、日志区占位
- 无需 `renderer-ready` 才显示窗口

### 5.5 风险与注意事项

- 保留 UI 形态，不要在此阶段重做产品界面
- 若前端用到 prototype 的本地 HTTP 协议，必须改为 Electron IPC bridge

## 6. 阶段 3：Electron 基础能力接入

### 6.1 目标

把正式桌面应用所需的非业务基础能力接入新壳。

### 6.2 任务

- 主窗口生命周期管理
- 关闭到托盘策略
- 托盘图标与托盘菜单
- 当前用户级开机自启
- 主菜单或系统菜单基础设置
- 应用设置读写基础能力
- 文件选择与目录选择对话框
- 打开目录、打开日志目录等壳层能力

### 6.3 产出

- 托盘可用
- 自启开关可配置
- 基础设置可保存
- 目录选择和打开目录可用

### 6.4 验收标准

- 点击关闭按钮后可按预期最小化到托盘
- 托盘菜单至少可执行“显示主窗口”“退出”
- 自启开关状态可持久化
- 应用重启后设置能恢复

### 6.5 风险与注意事项

- 不要把业务状态直接塞进托盘实现
- 自启必须优先走当前用户级方案

## 7. 阶段 4：自有存储体系建立

### 7.1 目标

建立 `YiboBattleSwitch` 自有存储目录与数据读写能力。

### 7.2 任务

- 确定用户数据根目录
- 建立 `config / library / backups / diagnostics / logs` 目录结构
- 实现 `settings.json` 读写
- 实现账号库目录初始化
- 实现账号元数据读写
- 实现备份目录初始化
- 实现诊断目录初始化
- 建立存储版本号与未来迁移入口

### 7.3 产出

- 本程序自有数据目录可自动初始化
- 基础配置和账号元数据可持久化

### 7.4 验收标准

- 新安装环境首次启动能自动创建所需目录
- 删除用户数据目录后重启仍能自恢复初始化
- 账号库结构不依赖 `NewBeeBox\battleCache`

### 7.5 风险与注意事项

- 这一阶段只是建立基础存储，不要急于接历史导入逻辑
- 要提前给存储格式留版本字段

## 8. 阶段 5：Battle.net 基础能力接入

### 8.1 目标

在不做正式切换前，先把 Battle.net 本地对象读取与写入能力独立出来。

### 8.2 任务

- 识别 Battle.net Launcher 路径
- 识别 WoW 安装目录
- 读取 `Battle.net.config`
- 写入 `Battle.net.config`
- 读取 Battle.net 相关注册表
- 写入 Battle.net 相关注册表
- 建立 Battle.net 当前状态快照模型
- 启动 Battle.net Launcher

### 8.3 产出

- Battle.net 文件与注册表的正式 `infra` 模块
- 当前状态快照能力

### 8.4 验收标准

- 可以单独读取当前 `Battle.net.config`
- 可以单独读取当前注册表认证相关值
- 可以生成一份当前 Battle.net 状态快照对象
- 可以从应用内正常启动 Battle.net Launcher

### 8.5 风险与注意事项

- 不要在这一步尝试盲目复刻完整切换
- 文件、注册表、路径逻辑必须和 UI、IPC 解耦

## 9. 阶段 6：普通权限进程治理落地

### 9.1 目标

把“不需提权直接杀进程”的正式流程做成独立模块。

### 9.2 任务

- 识别 Battle.net / Agent 相关进程
- 收集 PID、进程名、路径、用户归属等信息
- 实现优雅关闭流程
- 实现普通权限强制结束流程
- 实现按镜像名结束
- 实现按 PID 精确补刀
- 实现退出轮询确认
- 实现失败原因分型
- 实现统一结果结构
- 为后续切换事务暴露 `stopForSwitch()`

### 9.3 产出

- `battlenet-process` 模块
- 可单独运行的进程治理测试用例或诊断入口

### 9.4 验收标准

- 在普通权限下，能识别当前 Battle.net / Agent 相关进程
- 能优先尝试优雅关闭
- 在可控场景下能自动结束相关进程
- 无法结束时能返回明确原因，而不是模糊失败
- 有残留时不会误报成功

### 9.5 风险与注意事项

- 必须把“结束动作”和“退出确认”分开
- 不要把“taskkill 执行过”当作“进程已经退出”
- 不要在残留进程存在时继续下一步写配置

## 10. 阶段 7：正式切换事务落地

### 10.1 目标

把账号切换做成一条具备备份、回滚、诊断的正式事务。

### 10.2 任务

- 读取目标账号快照
- 校验目标账号完整性
- 切换前自动备份当前健康状态
- 调用 Battle.net 进程治理
- 进程全部退出后恢复目标配置
- 恢复目标注册表认证
- 必要时恢复附加本地状态
- 启动 Battle.net Launcher
- 记录切换日志
- 失败时回滚到操作前备份
- 通过 IPC 暴露 `switchAccount`

### 10.3 产出

- `switchAccount` 正式用例
- 切换成功、失败、回滚三种结果模型

### 10.4 验收标准

- 成功切换时，日志中能看到完整事务链路
- 切换失败时，不会留下半写入状态而无回滚
- 进程未退净时，切换必须中止
- 用户能看到明确失败原因和下一步建议

### 10.5 风险与注意事项

- 不要把“仅恢复配置”“仅恢复认证”继续放在主路径中
- 主路径必须围绕“完整切换事务”

## 11. 阶段 8：导入与诊断能力迁移

### 11.1 目标

保留 `NewBeeBox` 的研究成果，但把它收敛为“导入 + 诊断”两类次级能力。

### 11.2 任务

- 实现从 `NewBeeBox\battleCache` 导入历史账号
- 将导入数据转写为本程序自有账号格式
- 记录导入来源字段
- 实现当前状态诊断快照
- 实现最近两次快照对比
- 实现 Markdown 报告导出
- 在 UI 中把导入和诊断放入次级区域

### 11.3 产出

- `importFromNewBeeBox`
- `takeDiagnosticSnapshot`
- `compareLatestDiagnostics`

### 11.4 验收标准

- 卸载或删除 `NewBeeBox` 后，已导入账号仍可正常使用
- 导入完成后正式切换不再回读 `battleCache`
- 诊断报告可帮助分析切换前后文件/注册表差异

### 11.5 风险与注意事项

- 导入逻辑只能是一次性数据迁移，不得演化成运行时耦合
- UI 中不要把 `NewBeeBox` 放在主操作路径

## 12. 阶段 9：安装包与发布链落地

### 12.1 目标

建立正式桌面产品的构建、安装、版本与发布能力。

### 12.2 任务

- 建立安装包构建脚本
- 建立版本号管理流程
- 建立资源签名预留流程
- 确定安装目录与用户数据目录策略
- 生成标准安装包
- 预研或接入 `MSIX` 打包链
- 明确 Store 发布前置要求

### 12.3 产出

- 可安装的正式安装包
- 基础发布文档
- `MSIX` 准备清单

### 12.4 验收标准

- 安装包可在新机器上完成安装并启动
- 升级安装不会破坏已有用户数据
- 标准安装版运行时不依赖 `NewBeeBox`

### 12.5 风险与注意事项

- 安装目录不要承载账号库与日志主数据
- 发布链不能建立在原型脚本的临时打包方式上

## 13. 横向任务：测试与验收体系

这些任务贯穿所有阶段。

### 13.1 核心测试方向

- 冷启动与首屏稳定性
- 断网启动
- 用户数据目录初始化
- Battle.net 路径识别
- 配置文件读写
- 注册表读写
- 普通权限进程治理
- 切换成功路径
- 切换失败回滚路径
- 卸载 `NewBeeBox` 后独立运行

### 13.2 最少验收环境

- 已安装 Battle.net，且当前用户普通权限
- 有历史 `NewBeeBox` 数据的机器
- 无 `NewBeeBox` 的纯新环境
- 至少一台“能正常关闭进程”的环境
- 至少一台“会出现残留进程”的环境

## 14. 横向任务：日志与诊断要求

首期就必须具备较强的日志能力。

### 14.1 日志要求

- 主进程启动日志
- 渲染进程错误日志
- IPC 调用日志
- Battle.net 进程治理日志
- 切换事务日志
- 回滚日志

### 14.2 诊断要求

- 关键步骤都要可定位失败点
- 日志目录可从 UI 一键打开
- 用户报问题时可以导出最小必要诊断材料

## 15. 横向任务：对 NewBeeBox 依赖的持续审查

每阶段结束后都要做一次依赖审查。

检查项：

- 运行时是否仍读取 `NewBeeBox` 目录
- 是否仍引用 `nbb-core`
- 是否仍依赖 `NewBeeBox` 在线接口
- 是否仍把 `battleCache` 当主数据源
- 是否仍要求机器上必须存在 `NewBeeBox`

只要其中任意一项为“是”，就说明该阶段没有真正完成独立化。

## 16. 推荐里程碑

### 里程碑 A：新壳可运行

完成阶段 1 到 3。

成功标志：

- 新 Electron 应用能稳定启动
- UI 可显示
- 托盘、自启、基础设置可用

### 里程碑 B：独立数据层建立

完成阶段 4 到 5。

成功标志：

- 本程序自有账号库、配置、备份、诊断目录已建立
- Battle.net 文件与注册表读写能力可用

### 里程碑 C：普通权限切换闭环

完成阶段 6 到 7。

成功标志：

- 普通权限下完成大多数 Battle.net 关闭与切换事务
- 失败时可回滚、可解释

### 里程碑 D：独立产品成型

完成阶段 8 到 9。

成功标志：

- 仅保留 `NewBeeBox` 导入能力
- 正式安装包可发布
- 工程已具备后续 `MSIX / Store` 基础

## 17. 最终完成定义

本重建任务只有在以下条件同时满足时才算完成：

- 新应用运行时完全不依赖 `NewBeeBox`
- 当前 `electron-main.js` 历史大包不再参与正式程序
- UI 已迁入新 Electron 工程
- Battle.net 切换形成正式事务
- 普通权限进程治理可用
- 自有账号库成为唯一主数据源
- 标准安装包可交付

## 18. 一句话总结

这份任务清单的核心不是“按功能堆代码”，而是：

**先建立独立的新 Electron 产品骨架，再把 Battle.net 本地能力、普通权限进程治理、正式切换事务、自有账号库和发布链逐层接入，最终彻底摆脱 NewBeeBox 运行时包袱。**
